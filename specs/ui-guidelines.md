# UI ガイドライン

## 基本方針
- 共通色と基礎トークンは `src/app.css` の CSS 変数を使う。
- Tailwind を優先しつつ、画面全体のレイアウトや状態依存スタイルは既存 CSS と併用する。
- アイコンは `Material Symbols Rounded` を使う。
- 情報密度は高いが、主要操作は 1 画面内で視線移動が完結するように置く。

## メイン画面の考え方
- 発駅、区間、運賃サマリー、追加導線の順に上から下へ流れる構成を維持する。
- 編集対象である `mainRoute` を中心に据えるため、補助導線であるドロワーやオプションは主導線を遮らない位置に置く。
- 区間カードとサマリーカードは、どこまでの経路を対象にするかが一目で分かるようにする。

## エラーとメッセージ
- エラーは画面上部または対象セクション直下にインライン表示する。
- 一時成功メッセージも同じバナー系 UI に寄せ、通知手段を増やしすぎない。
- 画面遷移後にメイン画面へ返すメッセージは `mainScreenErrorMessage` を使う。
- 不正 URL や復元失敗などのケースは、クラッシュさせず文言付きで復帰導線を出す。

## ダイアログ
- 保存画面、発着駅選択画面、メイン画面ではページ内モーダルを使う。
- ブラウザ標準ダイアログに依存せず、文脈説明と確認アクションを同時に見せる。
- 破壊的操作では、対象が何かをダイアログ本文で明示する。

## リストとカード
- 保存済み経路、駅一覧、路線一覧、履歴は縦方向に素直に並べる。
- きっぷホルダのみ並べ替え可能リストとして扱う。
- カードはタップ可能領域を広く確保しつつ、内包する補助操作は誤発火しないよう分離する。

## きっぷホルダ UX
- PC は drag and drop、モバイルは touch 操作で並べ替える。
- モバイルの削除は誤操作防止のため、編集モードを経由する。
- `FarePicker` の操作はカード選択や詳細遷移を誤発火させない。

## テーマ
- ライト / ダークを実装済みで、結果は `localStorage.theme` に保存する。
- 色差だけに依存せず、罫線・背景面・アイコンで状態差を補う。

## ナビゲーション
- メイン画面と詳細画面の遷移は横方向スライドを基本とする。
- その他の画面は、操作速度と情報密度を優先して過剰な遷移演出を前提にしない。
- 下部固定ナビは主要アクションをまとめるが、スクロール時に主要情報を隠しすぎないようにする。
